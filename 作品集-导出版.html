一、项目背景

为什么做这个项目

作为应届毕业生，我对AI应用落地很感兴趣。在学习产品经理相关课程时，发现了一个明显的痛点：

日常学习中的困境
  • 产品经理岗位需要大量的文档练习（PRD、需求分析、功能设计）
  • 每次练习都要从头写，耗时且难以对比学习
  • 网上能找到的案例大多是"玩具级"Demo，缺乏完整的生产级实现

选择做这个项目的原因：

  1. 解决实际问题：自己学PM需要练手，这个工具能提升练习效率
  2. 技术契合度高：涉及AI、产品设计、前后端实现，适合作为学习项目
  3. 有差异化：不是简单的大模型套壳，而是围绕PM工作流的完整工具

项目定位

这是一个AI辅助PRD生成工具，面向产品经理学习者和初级PM，帮助他们快速完成从需求分析到PRD文档生成的完整流程。

三、核心功能设计

功能一：智能需求分析

功能描述：用户输入一段自然语言描述的产品需求，系统自动拆解为结构化的需求要素。

输入示例
做一个新能源充电桩预约系统，支持车主预约充电桩、查看充电状态、在线支付等功能

输出结构：

"用户角色": ["车主", "管理员", "维修人员"], "核心功能": ["充电桩查找与预约", "预约时间管理", "充电状态实时查看", ...], "用户流程": { "主流程": "查找充电桩 → 选择时间 → 确认预约 → 扫码充电 → 支付离开", "异常流程": "预约冲突处理、无空闲桩引导、改签/取消预约" }

设计思考：为什么拆解成这三个维度？用户角色明确服务对象，核心功能确定产品边界，用户流程梳理核心链路。这三个维度恰好对应PRD中的必备要素。

功能二：自动生成PRD文档

功能描述：基于需求分析结果，自动生成符合规范的PRD文档。

章节 内容 作用
产品概述 背景、目标、范围 让读者快速了解产品
用户分析 角色、场景、需求 明确服务谁
功能需求 功能描述、流程、规则 告诉开发做什么
非功能需求 性能、安全、兼容性 质量约束

设计原则：结构优先于内容；用户视角（每个功能从"用户故事"角度描述）；可验证（每个功能都有明确的验收标准）。

功能三：流程图自动生成

功能描述：基于用户流程描述，自动生成Mermaid格式的流程图。

// 输入：用户流程描述 graph TD A[开始] --> B[查找充电桩] B --> C{有空闲桩?} C -->|是| D[选择时间] C -->|否| E[查看附近桩] E --> B D --> F[确认预约] --> G[生成订单] G --> H[扫码充电] --> I[实时监控] I --> J[充电完成] --> K[在线支付] --> L[结束]

选择Mermaid的原因：技术可行性高（前端有现成的渲染库）；格式开放（非私有格式）；可扩展（支持导出PNG、SVG等）。

五、PRD设计能力

我理解的PRD

PRD（Product Requirement Document）是产品需求文档，是产品经理日常工作中最核心的输出物之一。一份好的PRD需要包含：

产品概述
→
用户分析
→
功能需求
→
非功能需求

PRD设计思路

在设计这个产品的PRD生成逻辑时，我参考了行业最佳实践，并做了简化：

  • 结构优先于内容：先有骨架，再填血肉
  • 用户视角：每个功能都从"用户故事"角度描述
  • 可验证：每个功能都有明确的验收标准

功能描述示例

❌ 错误示范（模糊）
用户可以查看充电桩状态
✓ 正确示范（具体）

功能：充电桩状态查看
角色：车主
场景：车主打开APP想找一个可用的充电桩
前置条件：用户已登录
主流程：进入列表页 → 系统展示附近充电桩（按距离排序）→ 每个桩显示位置、空闲桩数、价格 → 点击查看详情
验收标准：列表页加载时间<2秒；显示空闲桩数量准确；支持按距离/价格筛选

Prompt工程

Prompt设计是这个项目的技术核心。我设计了一套多角色Prompt体系：

// 不同角色负责不同任务 const PROMPTS = { analyzer: `你是一位资深产品经理，擅长需求分析...`, prd_writer: `你是一位专业的PRD文档专家...`, flowchart_generator: `你是一位UE专家，擅长绘制流程图...` };

Prompt设计原则：

  • 角色定义：明确AI扮演什么角色
  • 输出格式：严格要求JSON/Markdown格式
  • 约束条件：避免废话、要求数据驱动
  • 示例引导：Few-shot提升输出稳定性

作为产品经理，我清楚自己的定位：我负责产品设计、Prompt设计、用户体验；技术配合负责后端API、前端实现、部署运维。我不追求自己能写所有代码，但要求理解技术可行性能和开发有效沟通。

九、优化与反思

反思一：Prompt质量的挑战

问题：早期输出的PRD经常"假大空"，全是套话没有实质内容。

解决：增加约束条件"用数据说话，避免空话"；给Few-shot示例告诉AI什么是好的输出；收集Bad Case针对性加限制。

反思：Prompt调优是"炼丹"，需要耐心和系统性方法。

反思二：用户体验的平衡

问题：功能多了之后，界面变得复杂，新用户不知道从哪里开始。

解决：简化主流程（第一版只展示最核心功能）；增加引导（空状态时给出使用提示）；模板功能（让用户不用从零输入）。

反思：MVP思维很重要，不要一开始就想做"大而全"。

反思三：从产品经理视角看AI应用

收获：做这个项目让我真正理解了AI产品的本质：AI是工具而非产品；核心价值在于场景选择和输出调优；产品经理的竞争力在于判断AI输出质量和持续优化能力。

未来方向：引入用户评分体系形成反馈闭环；探索多模态能力（如手绘原型转代码）；针对垂直行业做深度定制。
